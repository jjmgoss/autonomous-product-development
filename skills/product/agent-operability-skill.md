# Product Skill: Agent Operability

## Goal

Determine whether a product can realistically be built, operated, and iterated on mostly through software and agent workflows.

## When to use

- during opportunity selection
- during validation
- before expanding MVP scope

## Procedure

1. Check build feasibility.
   - Are the requirements understandable and narrow?
   - Can the first useful version be built without a platform-sized architecture?
2. Check run-state feasibility.
   - Can the product operate mostly through software rather than human service delivery?
   - Does the MVP avoid physical operations, field work, or manual fulfillment?
3. Check support burden.
   - How much human handholding, trust-building, or custom setup would early users require?
4. Check dependency burden.
   - Does the wedge rely on fragile APIs, unavailable datasets, or hard-to-obtain integrations?
5. Check compliance and trust burden.
   - Is the MVP blocked by regulation, security review, or sensitive-data handling beyond what a narrow prototype should assume?
6. Record the limiting factor honestly.

## Strong signs

- clear narrow workflow
- software-only or information-only delivery
- low-touch onboarding possible
- manageable support burden
- simple data and integration needs for the first wedge

## Weak signs

- service-heavy onboarding
- bespoke implementation per customer
- strong need for enterprise approvals
- high compliance burden from day one
- unclear boundary between software product and human-operated agency work
